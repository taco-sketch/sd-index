메인 deck01 / 09
Opening

소셜디자이너의 활동을
입력, 해석, 판정, 브리프, 플랫폼 조회까지 연결하는 운영 인프라

통합모델과 통합시트는 단순 점수표가 아닙니다. 자료를 구조화하고, 사실을 추출하고, 판단 근거를 남기고, 외부 설명 가능한 브리프와 플랫폼 뷰까지 이어주는 운영 흐름입니다.

Core proposition

평가, 기록, 브리프, 외부 설명이 서로 끊기지 않는 구조

Raw Input → Fact Extraction → A/B/P → Gate → Brief → Platform View. 입력과 판단과 발신이 한 번의 흐름으로 연결됩니다.

점수 통합이 아니라 의사결정 흐름 통합 사실 추출 선행 근거·상한·보완 과제 동시 기록
이 자료의 역할
“복잡한 운영체계를 외부가 이해 가능한 언어로 번역하는 발표자료”
외부 파트너 심사자·평가자 운영 실무자 향후 플랫폼 협업자
문제 정의02 / 09
Why now

왜 이 체계가 필요한가

문제는 점수 그 자체보다, 입력–해석–판정–발신이 서로 끊겨 있다는 데 있습니다.

핵심 문제: 문서가 곧 점수가 되고, 기록과 설명이 따로 논다.
기존 방식의 한계

문서가 곧 점수가 되는 구조

  • 신청서와 인터뷰를 바로 평가해 말 잘하는 팀이 유리해짐
  • 기록이 많은 팀이 과대평가될 수 있음
  • 관계형·현장형 팀은 문서 부족 때문에 과소평가되기 쉬움
  • 평가표와 기록표, 외부 설명자료가 따로 놀아 업무가 중복됨
통합모델의 접근

입력–사실–판단–발신을 하나의 흐름으로 연결

  • 원자료를 먼저 등록하고 사실 추출을 거침
  • 활동 부재와 문서 부재를 구분함
  • 점수는 숫자만이 아니라 근거·상한·보완 과제와 함께 기록됨
  • 최종적으로 브리프와 플랫폼 뷰까지 연결됨
이 체계는 평가의 공정성만을 위한 보정장치가 아니라, 입력–해석–판정–기록–발신을 하나의 운영 흐름으로 묶는 구조입니다.
핵심 철학03 / 09
Integrated logic

통합모델의 기본 철학

통합모델은 점수 통합이 아니라 의사결정 흐름 통합을 지향합니다. A, B, P는 각각 다른 질문에 답하고, Gate A → Gate B → Gate P로 연결됩니다.

한 줄 요약: 하나의 점수표가 아니라, 세 가지 판단 체계의 연결 구조입니다.
A

선발·성장 판정

이 팀이 어떤 문제를 어떤 방식으로 풀고 있는지, 소셜디자이너로서 어떤 성장 상태에 있는지를 읽습니다.

B

거래·협력 언어 번역

이 활동이 구매, CSR, 투자, 협력의 언어로 번역 가능한지 판단합니다.

P

리포트 생산 가능성

이 내용을 4주 내 브리프나 리포트 형태로 전환할 수 있는지 봅니다.

Gate A: 이 팀은 누구인가 → Gate B: 거래 언어로 번역되는가 → Gate P: 지금 외부 전달자료로 생산 가능한가
판정 구조04 / 09
Gate criteria

각 Gate는 서로 다른 질문에 답합니다

핵심은 세 점수를 합산해 순위를 만드는 것이 아니라, 팀 정체성·거래 번역성·생산 가능성을 서로 다른 기준으로 판단하는 데 있습니다.

즉, A·B·P는 같은 질문의 세 버전이 아니라 서로 다른 판정 축입니다.
Gate A

선발·성장 판정

문제정의, 실험학습, 현장운영, 시민협력, 지속자원, 확산전이, 조직역량을 통해 성장 단계를 읽습니다.

S1 발굴형 S2 운영형 S3 확장형
Gate B

거래·협력 번역 판정

이 활동이 구매, CSR, 공공협력, 투자 언어로 실제 번역 가능한지 판단합니다.

패키지화 납품 신뢰도 측정·증빙
Gate P

리포트 생산 가능성 판정

이 팀을 4주 내 외부 전달 가능한 브리프나 리포트로 정리할 수 있는지 판단합니다.

증빙 준비도 납기 가능성 리스크 점검
Gate A는 “이 팀이 누구인가”, Gate B는 “이 활동이 거래 언어로 번역되는가”, Gate P는 “지금 설명 가능한 자료로 생산 가능한가”를 묻습니다.
작동 흐름05 / 09
Workflow

입력에서 판정까지의 작동 방식

원자료는 곧바로 점수가 되지 않고, Fact Extraction을 거쳐 평가 가능한 근거로 전환됩니다.

핵심 원칙: 활동 부재와 문서 부재를 구분한다.
01

자료 수집

신청서, 인터뷰, 발표자료, 증빙자료를 수집합니다.

02

Raw Input

원자료를 등록하고 ingest 상태를 관리합니다.

03

Fact Extraction

문장을 사건·행동 단위 사실로 구조화합니다.

04

A/B/P 정리

점수 근거, 상한, 보완 과제를 함께 정리합니다.

05

Gate 판정

Gate A/B/P와 stage track을 확정합니다.

06

Brief 생성

외부 설명 가능한 문장과 제안 구조로 전환합니다.

07

Platform View

마스터 원장과 조회용 뷰에 반영합니다.

워크북은 해석 구조, 통합시트는 축적 구조를 담당합니다.

Data structure

사례 진입08 / 09
Sample cases

샘플 사례로 보는 번역 방식

통합모델과 통합시트는 추상적 구조에 머무르지 않습니다. 유형이 다른 팀들을 통해 판별력과 번역 방식을 함께 보여줍니다.

사례는 팀 소개가 아니라, 모델의 판별력을 보여주는 증거입니다.
사례 1

동락점빵

생활권 밀도형 / 생활 인프라 복원형

농촌 식품사막화와 생활권 불평등 문제를 이동장터와 생활 인프라 복원 모델로 푸는 사례.

 사례 2

커피클레이

패키지·확산 전환형 / 거래 번역 가능형

커피박 재자원화, 시니어 일자리, 교육·확산을 결합한 패키지 전환형 사례.

 사례 3

스튜디오어중간

담론·콘텐츠·회복 플랫폼형 / 발굴형

청년 투병 서사, 회복 공간, 매거진, 포럼, 연구를 연결하는 콘텐츠 기반 플랫폼 사례.
사례 페이지는 단순 팀 소개가 아니라, “이 통합모델이 서로 다른 유형을 어떻게 다르게 읽는가”를 보여주는 2부 자료로 이어집니다.
